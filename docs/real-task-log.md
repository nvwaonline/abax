# 真实任务验证记录 #1：galaxy http-server 健壮性审计

按项目纪律（每个能力阶段配一次真实任务验证），用 CLI + JSONL 持久化驱动内核，
对 galaxy 主仓库 `src/server/http-server.ts` 做了一次多轮缺陷调查。
每轮是独立进程，状态靠操作日志重放恢复。

## 调查过程

**第 1 轮**：声明目标 + 4 个假设（malformed JSON→500、未知空间→500、请求体无上限、字段缺失→500）+ 4 条公理（其中 AX4 用强否定头：`field_validation_handled(?c) → ¬finding(missing_field_returns_500, ?c)`）。返回的溯因提示精确列出每个假设还缺哪些观察。

**第 2 轮**：阅读真实代码后断言 5 条观察事实（evidenceRefs 指向具体文件行号）。闭包自动判定：

- H1 malformed_json_returns_500 → **supported**（`readJson` 的 `JSON.parse` 无局部处理，外层 catch-all 一律 500）
- H2 unknown_space_returns_500 → **supported**（`store.getSpace` 抛错落入 catch-all）
- H3 unbounded_body_buffering → **supported**（请求体全量缓冲无大小限制）
- H4 missing_field_returns_500 → **refuted**（路由层对缺失字段返回 400，过宽假设被强否定头自动反驳）

**第 3 轮**：`record_result` 记录结论，证据链指向三个派生 finding。

**反事实测试**：撤回观察 O1 → 派生 finding 级联消失 → 引用它的结论 R1 级联消失 → 目标与 H1 自动回到 open。真值维护全链路正确。

## 对主仓库的可执行产出

http-server 三个确认问题：malformed JSON 应返回 400 而非 500；未知空间 id 应返回 404 而非 500（`getSpace` 异常需映射）；`readJson` 需要请求体大小上限。

## 发现的摩擦点

1. **（已修复）无法记录结论**：working memory 操作集缺少 `record_result` / `record_conflict`，调查闭环走不到最后一步。本次已补上。
2. **（已修复）未知操作的错误信息不可读**：崩在内部函数而不是说"未知操作 + 合法操作列表"。已改为批次执行前整体校验。
3. **目标/假设必须是全基原子**：表达不了"存在某个 finding"（∃x finding(x)）。本次靠"每个候选问题一个假设"绕过，可接受，但任务更开放时会变成限制。候选改进：非基假设（含变量的原子，被任意实例支持即 supported）。
4. **naf 条件在提示里不可见**（设计使然）：提示只列缺失的正文字，模型需要记得"断言 `parse_error_handled` 会消除 finding"。规则文本本身可见，目前够用。

## 结论

调查闭环（假设 → 提示 → 观察 → 判定 → 结论 → 可撤回）在真实任务上成立。
词汇漂移在 3 轮 5 谓词规模下未出现——需要更长的任务（二三十轮、由真实模型而非作者驱动）才能逼出来。

# 真实任务验证记录 #2：种子化链式审计（galaxy MCP 工具层）

第二次真实审计，重点检验跨任务经验复用：从记录 #1 的空间蒸馏经验胶囊，
种子化新空间后审计主仓库 `src/mcp/` 的错误处理。

## 调查过程

**第 1 轮**：`create-space` + `seed`——4 条规则（AX1–AX4）连同 9 个谓词的完整词汇表跨任务继承。

**第 2 轮**：模式假设 `finding(component=mcp_tool_handlers, kind=?k)`（"这个组件存在某种问题吗"）+
针对新组件的规则 AX5 + 3 条真实观察。闭包行为全部正确：

- 继承的 AX4 在新组件上触发（zod 全面校验 → 强否定头排除 missing-field 类问题）
- 继承的 AX2 正确地**没有**触发（MCP 层没有 catch-all，前提不满足——种子规则不会过度泛化）
- 新规则 AX5 推出 `finding(kind=errors_break_protocol)`
- 模式假设 supported 并列出支持实例

**第 3 轮**：`record_result` 落结论。

## 对主仓库的可执行产出

MCP 工具层一个确认问题：`src/mcp/server.ts:62` 的 registerTool 回调没有错误守卫，
handler 抛错（未知空间、llmClient 未配置、zod 失败）直接穿透为 MCP 协议级错误，
模型读不到错误信息、无法自我修正。建议包一层 guard 返回 isError 文本结果（galaxy-core 的做法）。

## 发现的摩擦点

5. **（已修复）节点 id 曾是 store 全局唯一**：`record_result` 用自然 id `R1` 时与记录 #1 的
   `R1` 冲突。智能体的自然习惯是每个气泡都从 G1/F1/R1 开始命名——全局唯一是平面 Map 的
   实现事故。已改为 id 按空间隔离（存储键 `spaceId/nodeId`），种子后缀逻辑随之简化。

## 结论

蒸馏/种子链路在真实任务上成立：规则可迁移且不过度泛化，词汇表继承消除了跨任务命名漂移的起点，
模式假设让"开放性排查"可以表达。每次真实验证仍在产出内核改进——验证纪律继续有效。

# 真实任务验证记录 #3：本地模型驱动审计（gemma-4-26b vs qwen3.6-35b）

第一次由非作者模型驱动演算板。任务：审计 galaxy 主仓库 `src/server/` 的错误处理。
夹具：`npm run verify:local-model`，自动记录转录与摩擦指标。

## 结果对比

| 指标 | gemma-4-26b-a4b | qwen3.6-35b-a3b |
|---|---|---|
| 轮次 | 30（耗尽上限，未完成） | 19（主动 done） |
| 工具错误 | **26** | 0 |
| 无效回复 | 0 | 5（空回复） |
| 读文件 | 3 | 8 |
| 走完调查闭环 | 否（working memory 为空） | **是**（3 个 finding + result + 自我反驳 2 个假设） |

**qwen 的审计质量出乎意料**：发现 run.ts 缺 uncaughtException/unhandledRejection 全局处理器（新发现，
作者审计未覆盖）、readJson 的 JSON.parse 风险（与记录 #1 一致，且正确注明"被顶层 catch 捕获但应更显式"）；
还正确反驳了自己的两个假设（认出代码是裸 Node http 而非 Express）。词汇全程零漂移（signatureWarnings=0）。

**gemma 完全卡死**：第 1 轮就把操作写成 `{"declare_goal": {...}}`（而非 `{"op":"declare_goal",...}`），
错误信息只说"unknown op"没给正确示例，于是它用同一个错误形状原样重试了 26 次直到轮次耗尽。

## 发现的摩擦点（全部已修复）

6. **操作形状不容错**：`{op_name: {...}}` 是模型最自然的形状猜测，现在自动归一化。
7. **错误信息缺正确示例**：模型无法从"unknown op"自我修正；错误信息现在附带完整示例操作。
8. **原子结构不校验**：gemma 的字符串原子 `"finding(x)"`、qwen 的坏 desired 曾产生 `undefined()` 目标
   静默进入 working memory；现在结构校验给出带示例的可读错误。
9. **夹具不兼容推理模型**：qwen 部分回合走 reasoning 通道导致 content 为空（5 次无效回复）；
   夹具现在回退读 reasoning_content、剥 `<think>` 块、token 预算提至 6000（可配）。

## 结论

核心论题首次得到外部证据：一个 35B 本地模型借助演算板完成了结构化、可追溯、能自我反驳的审计，
词汇零漂移——闭包引擎和签名警告承担了它心算最弱的部分。同时数据证明**错误信息的教学质量
直接决定弱模型的存活**：gemma 的 26 连败全部源于一条没有示例的报错。修复后值得重跑 gemma 对照。

# 真实任务验证记录 #4：gemma 修复后对照实验

记录 #3 修复（操作形状归一化、带示例的报错、原子校验）后用完全相同的任务重跑 gemma-4-26b。

## 对照结果

| 指标 | 修复前 | 修复后 |
|---|---|---|
| 轮次 | 30（耗尽，未完成） | **12（主动 done）** |
| 工具错误 | 26（同一错误循环到死） | **2（各犯一次，看到示例后立即改对）** |
| working memory | 空 | 目标 + 3 假设（1 个模式假设被实例支持）+ 2 观察 + 结论 |

教学闭环被证实：gemma 第一轮仍犯字符串原子和形状错误，但带示例的报错让它一次自我修正，
之后零重犯。"系统容错 + 报错教学"把 26B 模型从完全不可用拉进了可用区间。

## 与 qwen 的质量差距（能力边界数据）

gemma 完成了闭环但全程未写一条规则（axioms: none）——没有派生 finding，目标保持 open，
结论是散文式的；观察量也更少（2 vs 6 条事实）。次要发现：它的假设
`missing_error_boundary(function=?x, file=?x)` 把两个槽位写成同一个变量，意外要求 function=file 相等。

## 由此产出的改进（已落地）

10. **"无规则可推"提示**：gemma 的目标挂在 open 是因为没有任何规则的头能推出它，而溯因提示
    在这种情况下沉默。现在上下文会直接说："no rule derives this yet: add_axiom whose then
    matches it, or assert the fact directly"——把"该写规则了"这个步骤显式教给模型。

## 结论

三个本地模型实验（gemma 两次 + qwen 一次）建立了清晰的能力图谱：演算板对 35B 级模型即插即用，
对 26B 级模型在容错与教学修复后可用（但推理建模深度受限于模型本身）。
内核语义层在全部实验中零缺陷——所有摩擦都在输入接口与教学层，且全部可修。

# 真实任务验证记录 #5：gpt-oss-120b——协议纪律 ≠ 认知纪律

第三个模型，能力上限端。同一审计任务。

## 画像

11 轮主动完成；0 无效回复（JSON 纪律完美）；0 词汇漂移；4 次工具错误全部是
工具/操作两层混淆（把 record_result 当顶层工具、把 done 当操作），每次都自我修正。

## 关键发现：最强的模型是最懒的演算板用户

它只读 2 个文件、断言 0 条观察事实、写 0 条规则、无视全部三条 "no rule derives this yet"
提示，直接把"无问题"作为 finding 断言进板，宣布审计完成。其结论（run.ts 无错误处理问题）
与 qwen 经规则推导验证的发现（缺全局 uncaughtException/unhandledRejection 处理器）直接矛盾，
且 qwen 是对的。强模型对自己的心算过度自信，把演算板当汇报表格而非推理工具。

但注意：**板让这种偷懒留下了完整的、可机检的痕迹**——3 个假设全部 open、facts 区为空、
findings 全是断言而非推导。认知偷懒无法藏匿，这本身就是演算板的价值：它未必能强制纪律，
但能让纪律缺失变得可见、可审计。

## 由此产出的改进（已落地）

11. **findings 区显示来源**：断言的 finding 标 `[asserted, not derived]`，推导的标 `[derived]`——
    未经证明的主张和规则引擎背书的结论一眼可辨。
12. **夹具容错工具/操作混淆**：顶层调用 working-memory 操作名时自动包装进
    update_working_memory；未知工具的报错附带两层结构说明。

## 三模型能力图谱（最终）

| | gemma-26B | qwen-35B | gpt-oss-120B |
|---|---|---|---|
| 协议纪律 | 修复后可用 | 良好 | 完美 |
| 方法论遵守 | 浅（无规则） | **完整** | 绕过 |
| 审计结论质量 | 浅但无错 | **最佳（发现真问题）** | 有错（漏报） |
| 板上痕迹可审计性 | ✓ | ✓ | ✓（偷懒可见） |

最佳用户不是最强模型而是中间档——强模型需要的不是容错而是纪律约束（SKILL/提示词要求
finding 必须 derived、结论必须引用观察证据），这是建模技能的下一轮输入。

# 真实任务验证记录 #6：大型真实代码库审计（olapdb 282 文件 / 5.4 万行 / 6 模块）

首次大规模实测：qwen3.6-35b 审计一个开发中的 Java OLAP 数据库内核（并发 bug、资源泄漏、
吞异常、状态不一致）。同模型、同任务、同 60 轮预算下跑了两次，天然构成 nudge 修复的对照。

## 对照结果（轮次经济 nudge 修复前 vs 后）

| 指标 | 第一次（nudge 前） | 第二次（nudge 后） |
|---|---|---|
| 写板批次 | 2 | **12** |
| 操作数 | 10 | **101** |
| 板上事实 | 0（全程空板） | **56** |
| 落结论 | 无 | **完整四类审计报告** |
| 读文件 | 49（无记账空读） | 25（读完即记账） |
| 签名警告 | 0 | 7（词汇基本稳定） |

同一模型，加上每轮 `[turn N/MAX]` 水位 + "连续 N 轮未写板"提醒后，从"读 49 文件零留存"
变成"56 条带行号事实 + 分四类的审计报告"。这是滑动窗口下"历史消亡、板持久"机制 +
轮次经济教学的直接证据。

## 断言真实性抽查（人工核实源码）

**准确**：RunnerManager.java:296 空 catch（region 关闭中断 runner 线程失败被吞）；
5 处 synchronized(X.class) 类字面量锁；CubeEnumerator queryCount 用
setQueryCount(getQueryCount()+1) 非原子自增（4 处，真实并发隐患）；多处段处理 Runner 空 catch。

**误报**：声称 CubeEnumerator scanner "任何路径都不 close" —— 实际有 closeScanner() 且
实现 AutoCloseable；声称 IngestBatchDedupe.reserve() 有 read-modify-write 竞态 —— 实际有
3 次重试循环 + insertReserved 原子插入。两个误报都源于"未读到底层实现就断言"。

## 关键发现：这次审计跳过了演绎层

56 条事实**全部 [asserted]，零 [derived]**。qwen 跳过了"观察→规则→闭包推出 finding"，
直接把怀疑当观察记账。后果是双重的：

1. 板退化成结构化笔记本——但即便如此，"记账 + file:line 证据"已把模型从
   "读 49 文件零产出"拉到"可复核审计报告"。增益层级 1（防遗忘/防拖延）和
   2（证据可复核）独立于演绎层就已生效。
2. **两个误报恰恰是跳过演绎层的代价**。若按 SKILL 纪律强制 finding 必须 derived，
   "scanner 不 close"必须先断言观察事实 `no_close_call(file=CubeEnumerator)`，
   而模型一读代码就会发现 closeScanner() 存在，该观察立不住，误报在规则触发前就被挡下。
   **误报根源 = 跳过演绎层，而非模型能力不足。**

## 三层增益模型（本系列最终结论）

演算板对模型能力的提升不是单一的，至少分三层，可独立触发：

- **层 1 记账**：把心算外化为持久事实，防遗忘、防漂移、防拖延。本次贡献最大，
  把 49 文件空读变成 56 条留存事实。
- **层 2 证据**：file:line 可复核 + asserted/derived 标记，让真问题与误报可被人或规则区分。
- **层 3 演绎**：规则闭包做确定性推导。本次未触发——这正是误报未被挡下的原因。

它不提高模型智力，提高的是智力的**兑现率与可审计性**。强制演绎层（SKILL 纪律）是下一轮
把误报率压下去的关键。

## 由此产出的改进（已落地）

13. **夹具崩溃安全**：LLM 调用 3 次重试（间隔递增）；致命错误仍强制落盘 final board + metrics，
    不再丢失整轮结果（第一次大型运行曾因此只剩部分日志）。

## 结论

演算板在真实大型代码库（282 文件）上成立：上下文有界（滑动窗口下任务长度无上限）、
词汇近乎稳定（7 警告 / 101 操作）、产出可复核的真实情报（含至少 3 个值得修的真问题）。
同时诚实暴露了边界：跳过演绎层时误报率上升。这不是反例，而是对"层 3 为何重要"的实证。

# 真实任务验证记录 #7：演绎层引导对照实验（同 olapdb 任务）

记录 #6 发现"qwen 跳过演绎层、56 条全 asserted finding"后，加入三项引导（系统提示嵌入
可照抄的 observation→rule→finding 模板、检测到 asserted finding 时的演绎 nudge、SKILL 同款模板）
并新增 derivedFindings/assertedFindings/rules 三个结案指标，用完全相同任务重跑。

## 对照结果（引导前 vs 后）

| 指标 | 记录 #6（引导前） | 本次（引导后） |
|---|---|---|
| derived findings | **0** | **7** |
| asserted findings | 56 | **0** |
| 规则数 | 0 | **7** |
| 目标 | 全 open | **satisfied** |
| 轮次 | 58 | 23（更快收口） |

行为被彻底翻转：从"断言堆砌"变成"全部经规则推导、目标判定满足"。**层 3 演绎首次被触发**，
系统提示里的可照抄模板被直接采纳——`ax_swallow: IF empty_catch(file,line) THEN
finding(type=swallowed_exception,...)` 是教科书式的正确演绎，4 个真实空 catch 观察
（Util:713、RunnerManager:296、SegCombineRunner:56、SegMendRunner:30，均人工核实为真）推出 4 条 derived finding。

## 更深的发现：演绎的形式 ≠ 演绎的实质（同义反复规则）

引导成功逼出了演绎的形式，但暴露了一个新现象。两类规则质量截然不同：

- **真演绎**（swallowed_exception）：规则体 `empty_catch(file,line)` 是比结论更原始、
  可独立 grep/定位的具体特征。误报会被挡——要触发就得先断言一个能被核实的观察。
- **同义反复规则**（resource_leak / concurrency_bug / inconsistent_state）：qwen 造了
  `IF suspected_resource_leak(...) THEN finding(type=resource_leak,...)`——规则体与头是同一件事
  换皮。这种"演绎"没有任何过滤力，误报（Cluster.java line=N/A、未读底层即判断）照样
  以 [derived] 身份溜进 findings。

这是记录 #6"两个误报"的精确化结论：**演绎层的过滤价值取决于规则体是否是比结论更原始、
更可独立观察的事实**。`empty_catch` 是（可机械观察）；`suspected_concurrency_bug` 不是
（它就是结论本身）。纪律 nudge 能逼出演绎的形式，但模型会在"难以机械观察"的抽象问题类别上
用同义规则规避实质。

## 由此产出的改进（已落地）

# 真实任务验证记录 #24：算术内建可学性——模型一次照抄模板即用对

- cost(widget,48063) 与 cost(gadget,47064) 均标 **[derived]**——闭包推出，非模型断言。
- 规则照抄模板成功：`IF line(item=?i,unit=?u,qty=?q) AND mul(left=?u,right=?q,result=?t)

  （没退化成两条硬编码）。
- 数值精确（1299×37=48063、888×53=47064），但**关键不是数值对，是 [derived] + 规则含 mul**——

正面回答"演算板新增能力，模型够不够知识去用"：**给一个能照抄的具体模板就够**——与全系列

# 真实任务验证记录 #25：动作/反应可学性——模型一次用对消耗-生产，且没混淆规则与动作

承 #24，验证更难一档的能力：化学反应 2H2+O2→2H2O 建成 define_action（消耗反应物、生产水），
simulate 后 apply。这比算术多两层：动作是两步（定义+提交），且负效果"消耗"语义需要理解
"规则单调删不掉、必须用动作"。

## 板证（变换真的发生）

- actions 区有 burn：`PRE have(H2) AND have(O2); EFFECT ¬have(H2) AND ¬have(O2) AND have(H2O)`
  ——负效果消耗、正效果生产，语义用对。
- apply 后 **facts 区只剩 have(H2O) [derived]**——H2/O2 被真实删除（消耗），不再在板上。
  这是非单调变换确实执行，非散文声称。
- goal have(H2O) [satisfied]（闭包判定）。

## 关键正面信号：没犯"用规则做消耗"的概念错误

最可能的概念混淆是把消耗写成 add_axiom（但规则单调，删不掉前提）。模型**正确区分了两层**：
用 production rule 还是 transformation action，选了后者。这说明提示里"rules are monotonic -
they never delete; actions do"这句教学被吸收了——又一次印证：把概念差异**显式写进可照抄模板**，
模型就能正确选择计算模型。

## 小保留（不影响结论）

模型建成**布尔反应**（have/无，未带 mol 计量）——任务的合法解（未要求化学计量）。带计量的
版本（#15 量化反应测试证明内核支持）模型是否也能一次用对，是更强测试，留待后续。另：板留状态
不留过程，无法从终态判断它是否真的先 simulate 再 apply（需翻转录）——与算术轮同样的可观测性限制。

## 综合结论（#24+#25）

**演算板新增的两类能力——产值算术、消耗-生产动作——对模型均可学，条件一致：给可照抄模板 +
把概念差异显式写进提示。** 这把"能力存在但模型看不见/不会用"的可发现性缺口彻底闭合，并第三次

# 真实任务验证记录 #26：计量反应可学性——模式泛化级通过，并暴露两个真机才抓得到的内核缺口

承 #24/#25 的"两类新能力可学"结论，本轮测更难的组合：化学计量反应（5 mol H2 + 3 mol O2，
2H2+O2→2H2O 按反应单元推进），要求绑定当前量、gte 守卫、sub/add 算新量、效果换事实，
simulate 后反复 apply 至反应物不足。提示模板只给了 H2+H2O 两个物种的形态。

## 模型侧：通过，且超出照抄

- **模式泛化**：模板只示范两物种，模型自行把"绑定+守卫+算术+换事实"四件套扩展到 O2——
  全系列首次观察到超出模板照抄的结构泛化（此前 gemma/qwen/27B 均为照抄级）。
- **数值精确**：两次应用 (5,3,0)→(3,2,2)→(1,1,4)，第三次被 gte 挡停。
- **新诊断被吸收**：结案理由"H2 (1 mol) < required 2 mol"正是本轮新加的
  first-failing-precondition 诊断（gte(left=1, right=2) 代入值）的转述——报错教学链路打通。

## 内核侧：板证暴露两个缺口（沙盒全绿，真机现形）

1. **消耗事件被消耗本身藏掉**：apply 事件的 evidenceRefs 引用了被消耗（归档）的事实，
   而可用性沿证据链递归判定 → 事件自判不可用，从板上消失。板上 results 只剩模型自己的
   record_result，过程痕迹全无。沙盒测试只断言了节点存在（store.getNode），没断言板上可见
   ——测试盲区即缺口所在。修复：事件只引用 action 节点，消耗/产出/归档 id 进摘要文本
   （历史记录的有效性不得依赖被记录物保持活跃）。
2. **推导门可被动作洗白**：isDerivedFactNode 把 createdBy=system 一律判 [derived]，
   动作效果事实因此被门当作"闭包背书"——模型本可把 finding(...) 写进效果、apply 一下绕过
   "finding 必须由闭包推导"的核心不变量（#26 板上三个动作产物全标 [derived] 即此症状）。
   修复：该判定收紧为闭包专属（derived: id 前缀 / 闭包摘要前缀），动作产物板上改标
   **[effect]**——三种标记自此语义分明：[derived]=闭包背书，[effect]=动作产物，[asserted]=裸断言。

红测试先行（4 红，含两次 apply 连环消耗的精确失败形态），修复后 157/157，提交 0d818c4。
同轮顺带：用户首次本机跑 npm test 暴露 insideRoot 围栏的 Windows 路径缺陷（POSIX 夹具 +
已解析路径对未解析根串比较），修复为 path.relative 容器判定（57bfae4）。

## 结论

- **计量反应对模型可学，且达到模式泛化级**——"演算板三类能力（算术、布尔变换、计量变换）
  皆可学"完整成立，条件始终一致：可照抄模板 + 概念差异显式入提示 + 报错带教学。
- **"沙盒全绿 ≠ 正确"第三次应验**（继 #3 报错教学、围栏 bug 之后）：事件可见性缺口的测试
  恰好缺了"板上可见"这一断言——验证的盲区与实现的盲区同源。真机验证不是仪式，是唯一能
  对齐"测试以为的正确"与"使用中的正确"的手段。

# 真实任务验证记录 #28：硬难度算术 A/B——板 8/10 vs 裸跑 0/10，十题零错误主张

## 设定

GSM-Hard 式发票题：5–8 行，每行 5 位数量 × 8 位单价（积 13 位），精确总和。
种子 11，N=10，两臂同题同模型，`bench-arith` 夹具（随公开仓发布，可复现）。
计分从板不从散文：每条线积与总和必须以精确值出现为 `[derived]` 事实才算解出。

配置（config 头落盘）：qwen/qwen3.6-27b（官方，无思考通道），LM Studio 流式 SSE，

`totalAssertedOnly` 全程 0（无一例"数字对但仅是裸断言"）。

## 三个裁决

1. **主张精度不对称（核心结论）**：十题中 baseline 交出 3 个自信的错误数字；
   板上自始至终**没有出现过一个错误数字**——要么全链 `[derived]` 的精确值，
   要么（两次）什么都不主张。错而自信 vs 宁缺毋假。
2. **难度上行后心算从"不可靠"恶化为"不可终止"**：P4 起 baseline 七连 600s DNF
   （流式确认 token 持续产出——纯认知性不收敛，复读循环）。
3. **板的两场失利全在生成层，零推理层失误**：P5/P9 死于写 JSON 时复读，板上
   没有任何错误推导。取证回放（容错+逐轮转录）两题分别 3、4 轮干净解出，
   全部数值 BigInt 独立复核精确——**复败不复现，能力面无缺口**，属随机生成
   故障（该轮发生率 ~20%），单轮超时容错已覆盖该类。回放转录另证：模型自发
   增量建模（目标+事实 → 乘法规则 → 求和规则 → 收工），8 原子 + 7 级链式 add
   依赖序一次写对。

## token 经济学（12.5 tok/s 折算）

baseline 成本**计算性**增长（随位宽×行数）且带不收敛风险（600s ≈ 7,500 token
预算内 0 次成功）；board 成本**结构性**固定（N 事实 + 2 规则 ≈ 2–3K token，与
位宽无关）。P1 对照：~2,400 token 换一个错值 vs ~2,600 token 换精确值+证据链。
板臂账单全是机械成本（随硬件/MTP 缩放）；baseline 的不收敛不随硬件缓解。

## 过程修复台账（本轮基建产出，全部红转绿）

1. LLM 客户端默认流式：非流式下 undici headersTimeout≈300s 掐死长生成（首轮
   全部 baseline "DNF" 的真因——基建假阳性，已重测）。
2. SSE 累积式 message chunk 改替换语义 + 流量保险丝（O(n²) 活字符串 → 4GB OOM）。
3. 重试政策：预算裁决（超时/流帽）不重试，瞬态故障才重试。
4. 板臂单轮超时容错（吸收一次）+ 崩溃取证（部分板照计分、boardDump 入日志）。
5. bench 日志 config 头（模型归属取自 env 非 UI 横幅——#27 教训制度化）+
   GALAXY_BENCH_NOTE 标注 + 失败行 boardDump + GALAXY_BENCH_TRANSCRIPT 逐轮转录。
6. **内核缺口：连接爆炸防护**。mtp 预跑 P2 板臂 4GB OOM 炸进程——求和规则不钉
   item（8 全自由变量原子 × 8 事实 = 8⁸ ≈ 1670 万绑定）时 matchConditions 无上限。
   修：join 宽度帽（GALAXY_MAX_JOIN_BINDINGS，默认 10 万）增量检查 + 点名规则的
   教学报错（"钉死标识参数或跨原子复用变量"）。真机基准再次完成本职：暴露沙盒
   永远撞不到的内核缺口。
